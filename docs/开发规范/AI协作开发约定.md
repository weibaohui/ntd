# AI 协作开发约定（AI Development Contract）

> 本文档用于 **约束与指导 AI 参与 ntd（Now Task, Done）项目代码编写时的完整协作流程**，
> 确保代码质量、可维护性与团队协作一致性。
>
> **适用对象**：所有参与本仓库开发的 AI Agent（含 Claude Code、Copilot 类、自动化 Agent）。
>
> **本文件链接**：被 `AGENTS.md` 中的步骤 2 引用，AI 在开始工作前必须读取。

---

## 0. 总体原则（必须遵守）

1. **AI 是协作开发者，而不是直接提交者** — 所有 PR 的最终合入权在人类
2. **AI 必须遵循与人类开发者一致的工程规范**
3. **所有 AI 行为必须可追溯、可审查、可复现**

---

## 1. 工作原则

### 1.1 写代码前必须确认设计方案（强制）

**在动手写任何代码之前，必须先向人类说明设计方案并得到明确确认，方可开始实现。**

#### 1.1.1 要求

1. **描述方案**：说明打算做什么、改哪些文件、核心逻辑是什么
2. **等待确认**：收到人类明确同意（如"好"、"可以"、"proceed"）后才开始写代码
3. **禁止自作主张**：不得在未经确认的情况下直接输出代码或修改文件

#### 1.1.2 适用场景

- 新增功能或命令
- 重构现有代码结构
- 涉及多个文件的改动
- 任何架构层面的决策

#### 1.1.3 例外

以下情况可直接执行，无需额外确认：
- 人类已在指令中描述了具体实现细节
- 修复明确的单行 bug（如语法错误、typo）

---

### 1.2 遵循 YAGNI 与极简原则

在开始编码前，必须先执行以下反思检查：

- **YAGNI**：这东西真的需要存在吗？（不需要就跳过）
- **标准库**：标准库有吗？（有就用）
- **原生特性**：浏览器/平台原生支持吗？（有就用）
- **依赖**：已安装的包里有吗？（有就用）
- **极简**：能一行搞定吗？（一行搞定）
- **最后手段**：只写能跑通的最小代码

---

## 2. 开发前置约定（Before Coding）

### 2.1 必须读取仓库约定文档

在开展任何工作之前，AI **必须先阅读并理解**以下内容（如存在）：

- `README.md`
- `AGENTS.md`
- `CLAUDE.md`
- `docs/开发规范/AI协作开发约定.md` —— 本文档
- `docs/开发规范/后端规范/` 下的后端规范
- `docs/开发规范/前端规范/` 下的前端规范
- 相关需求文档（如 `docs/requirements/`）

> ❗ 禁止在**未读取仓库约定**的情况下直接开始写代码。

### 2.2 必须基于分支开展工作

- 所有开发必须通过功能分支进行，**不区分 AI 或人类**
- **禁止在 `main` 分支上直接修改代码**
- 分支命名规范：

```text
feat/xxx       # 新功能（同本项目约定，替代 feature/）
fix/xxx        # 缺陷修复
refactor/xxx   # 重构优化
chore/xxx      # 工程性调整
docs/xxx       # 文档变更
```

- 所有代码修改 **仅允许发生在对应分支下**

### 2.3 模糊需求必须先澄清（grillme 流程）

当收到简短或模糊的需求描述时，AI **不得直接开始写代码或写文档**，必须先执行需求澄清流程：

#### 步骤一：分析现状

阅读相关代码、配置和已有文档，了解当前系统在该功能区域的实际实现情况。

#### 步骤二：整理提问清单

将需要澄清的点组织为选择题，每个问题给出 2-4 个选项（A/B/C/D），让用户通过选择来明确意图：

```text
## 需求澄清

我已阅读了相关代码，当前系统的 Todo 列表工作方式为：
- 支持分页查询，每页 20 条
- 排序方式为 created_at DESC
- 不支持关键词搜索

关于你的需求，我有以下几个问题：

1. 分页大小
   A) 保持 20 条/页
   B) 改为用户可配置（默认 20）
   C) 改为无限滚动
   D) 不做分页，一次加载全部

2. 排序方式
   a) 保持 created_at DESC
   b) 改为手动拖拽排序
   c) 支持多种排序切换（创建时间/截止时间/优先级）

请回复 1a2b 这样的格式选择。
```

- 选项用字母 `A/a`、`B/b`、`C/c`、`D/d` 标注
- 每个问题独立编号（`1.`、`2.`、`3.`）
- 要求用户回复 `1A2C3B` 或 `1a2c3b` 格式
- 用户回复后，AI 将选择转化为需求文档的明确条目

#### 步骤三：确认后进入文档流程

用户回复选择后，将确认结果写入需求文档，再进入设计、编码阶段。

> **目的**：防止"我以为你懂了"导致的返工，用结构化提问帮用户把需求说明白说具体。

---

### 2.4 写代码前必须先写文档

在编写任何代码之前，AI **必须先编写需求文档、设计文档、测试文档**，用于明确：

- 要解决的问题
- 功能目标与边界
- 核心设计思路
- 可能影响的模块
- 修改文档必须填写变更记录表
- 测试用例必须符合 `docs/文档编写规范/` 中的约定

#### 文档存放规则（强制）

- 文档 **必须放在约定的文档目录中**：
  - 需求文档 → `docs/requirements/`
  - 设计文档 → `docs/design/`
  - 测试文档 → `docs/testing/`
  - 缺陷记录 → `docs/bugs/`
  - 架构决策 → `docs/decisions/`

- ❌ 不得随意将文档放在根目录或代码目录
- ✅ 遵循 `docs/README.md` 和 `docs/文档编写规范/README.md` 中的约定

#### 变更记录表

所有修改文档必须在文件头部增加变更记录表：

```markdown
| 修改人 | 修改时间 | 修改内容 |
|--------|---------|---------|
| <作者> | YYYY-MM-DD | 初始版本 |
```

## 3. 编码约定（During Coding）

### 3.1 注释与可读性要求

**本项目的注释要求极高，必须严格执行 CLAUDE.md 中的注释规范：**

- **逐行注释**：每一行代码都要写注释，解释「为什么这么写」（不是「写了什么」）
- **段落总览注释**：在大段代码之前先用一段注释说明整体处理思路、输入输出和关键步骤
- **禁止随意删除已有注释**
- 如确需删除或调整注释，必须显式标记：

```text
// 原注释内容（删除，待确认）
```

> 目标：即使 AI 不在场，人类也能理解代码为什么这么写。

### 3.2 函数长度与测试要求

- 单个函数体（不含函数签名、空行、注释）不得超过 **30 行**
- 超过 30 行的函数必须拆分为多个职责单一的子函数
- 避免深层嵌套，超过两层嵌套应抽取为独立函数
- **每一个公开函数/方法必须有对应的单元测试**
- 测试必须覆盖正常路径、边界条件以及预期的错误处理分支

### 3.3 API 与接口一致性

- 在编写涉及接口调用或接口实现的代码时：
  - 必须检查是否存在对应的 API 文档
  - 严格遵循接口定义（字段、语义、错误码）
  - 涉及路由变更时必须同步更新路由注册

- ❌ 禁止在未同步更新文档的情况下擅自扩展或修改接口行为

### 3.4 编译告警清理

- Rust 后端：`cargo clippy --all-targets -- -D warnings` 必须零告警零错误
- 生产代码禁止 `.unwrap()` / `.expect()` / `panic!`（`#[cfg(test)]` 和 `build.rs` 除外）
- 前端：`npx tsc --noEmit` 必须零错误
- 因存量代码引入新告警时，必须先修复存量告警再合入新代码

### 3.5 日志输出要求

- 使用合适的日志框架输出（后端使用 `tracing` / `log` crate）
- 关键业务路径必须包含足够的日志以便问题排查
- 生产环境日志级别应适当，避免输出敏感信息

## 4. 完成后的强制检查（After Coding）

### 4.1 编译与测试

代码完成后，AI **必须执行并确认**：

- 项目可正常编译 / 构建
- 已有单元测试可通过（`cd backend && cargo test`）
- 新增功能应补充必要的单元测试
- 无编译警告（`cargo clippy --all-targets -- -D warnings`）

### 4.2 功能测试（必须）

AI 在完成代码后，**必须对所写代码进行一次功能测试**，包括但不限于：

- 对修改的 API 进行请求验证
- 对 UI 修改进行 Playwright 验证（`cd frontend && npx playwright test --reporter=list`）
- 按 `docs/文档编写规范/` 中的约定执行

### 4.3 安全扫描与安全反思（必须）

AI 在完成代码后，**必须对所写代码进行一次安全自检**，包括但不限于：

- 是否存在：
  - 明显的逻辑漏洞
  - 越权访问风险（如 workspace 层级权限检查缺失）
  - 输入未校验问题（SQL 注入、XSS）
  - 敏感信息泄露风险
- 是否引入了：
  - 不必要的高权限操作
  - 不安全的默认配置

> 若无法自动扫描，必须以**文字形式进行安全反思说明**。

## 5. 功能总结与需求对应

### 5.1 功能总结文档（强制）

每一个功能开发完成后，AI **必须撰写功能总结文档**，内容包括：

- 实现了什么
- 与需求的对应关系
- 关键实现点
- 已知限制或待改进点
- 实现总结必须能明确对应到原始需求文档

### 5.2 存放规则

遵循 `docs/文档编写规范/README.md` 中的约定，存放在：
`docs/requirements/<编号>-<功能名称>-实现总结.md`

---

## 6. 缺陷处理

### 6.1 缺陷修复记录（强制）

每个缺陷都必须记录在 bugs 目录中，结构如下：

```text
docs/bugs/
  <编号>-<缺陷标题>/
    <编号>-<缺陷标题>-缺陷说明.md
    <编号>-<缺陷标题>-缺陷分析.md
    <编号>-<缺陷标题>-修复总结.md
```

遵循 `docs/文档编写规范/Bug缺陷处理文档编写规范.md` 中的约定。

---

## 7. 提交与审核

- ❌ **AI 不得自行决定代码是否合入 main 分支**
- AI 负责编码与说明，人类负责最终审核与 PR 合入
- PR 创建时应包含：
  - 实现了什么
  - 与需求的对应关系
  - 关键实现点
  - 测试与验证结果
  - 已知限制或待改进点

---

## 8. 违约处理原则

若发现以下情况，AI 应立即停止继续编码，并提示人类介入：

- 无法确认仓库规范
- 文档目录或需求目录不清晰
- 接口定义存在冲突或歧义
- 关键安全逻辑不明确
- 编译/测试存在无法解决的失败

---

## 9. 目标总结

> 这份约定的目标不是限制 AI，而是：
>
> - 让 AI 像一个**成熟、可协作的工程师**一样工作
> - 让 AI 的产出 **可维护、可审计、可持续演进**
> - 降低 AI 引入后的工程风险，而不是制造新的不确定性

---

**AI 必须默认遵守本约定，除非人类明确声明例外。**
