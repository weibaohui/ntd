# 后端规范 13：禁止清单

> 明确定义后端代码中**严禁出现**的代码模式。
>
> 此清单是编译告警之外的硬性规则，AI 必须逐条检查后方可提交代码。

---

## 代码模式禁止

| # | 禁止项 | 原因 | 替代方案 |
|---|--------|------|---------|
| 1 | `.unwrap()` / `.expect()`（生产代码） | 导致 panic | `?` 运算符 + `AppError` |
| 2 | `panic!()`（生产代码） | 导致进程崩溃 | 返回 `AppError` |
| 3 | `unsafe` 代码块 | 破坏 Rust 内存安全保证 | 使用 safe 抽象 |
| 4 | SQL 字符串拼接 | SQL 注入风险 | SeaORM Query Builder / 参数绑定 |
| 5 | 硬编码密码/Token | 安全泄露 | 配置文件 / 环境变量 |
| 6 | `.clone()` 循环内调用 | 性能问题 | 使用引用或 `Arc` |
| 7 | `async fn` 中阻塞调用（`.blocking().await` 误用） | 线程池饥饿 | 使用 `tokio::task::spawn_blocking` |
| 8 | `impl From<...> for Response` 之外的 HTTP 响应构造 | 架构混乱 | 统一使用 `AppError::IntoResponse` |

---

## 架构禁止

| # | 禁止项 | 原因 | 替代方案 |
|---|--------|------|---------|
| 9 | Handler 中直接调用 SeaORM | 职责越界 | 通过 Service → Repository |
| 10 | Service 引用 `axum` / `http` crate 类型 | 层间耦合 | 使用纯 Rust 类型 + DTO |
| 11 | Handler 中控制事务边界 | 事务应在 Service 层 | Service 方法内部 begin/commit |
| 12 | 跨模块循环依赖 | 编译错误、架构异味 | 抽离公共类型到 Model/共享模块 |
| 13 | 在 `#[cfg(test)]` 之外使用 `#[allow(...)]` | 掩盖告警 | 修复代码而非抑制告警 |

---

## 文件与提交禁止

| # | 禁止项 | 原因 | 替代方案 |
|---|--------|------|---------|
| 14 | 合入有编译警告的 PR | 零警告策略 | `cargo clippy -- -D warnings` |
| 15 | 未通过测试就提交 | 质量门禁 | `cargo test` 通过 |
| 16 | main 分支直接提交 | 分支策略 | 功能分支 → PR → 审核 → 合入 |
