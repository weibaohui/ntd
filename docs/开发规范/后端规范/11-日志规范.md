# 后端规范 11：日志规范

> 定义日志记录规范。

---

## 1. 日志框架

使用 `tracing` crate 作为主要日志框架：

```rust
// tracing 提供结构化日志和 span，适合异步环境。
// log crate 的宏（log::info!）仍然可用，但推荐统一使用 tracing。
use tracing::{info, warn, error, debug, span, Level};
```

---

## 2. 日志级别

| 级别 | 使用场景 |
|------|---------|
| `error!` | 不可恢复的错误、异常、业务失败 |
| `warn!` | 需要关注的异常但可以恢复（重试、降级） |
| `info!` | 关键业务流程节点（创建、删除、状态变更） |
| `debug!` | 调试信息（请求参数、数据库 SQL） |
| `trace!` | 详细追踪（循环迭代、函数进入/退出） |

---

## 3. 日志内容要求

```rust
// ✅ 好的日志：包含上下文信息、关键参数、操作结果。
// 记录"做了什么"和"结果如何"，方便问题排查时还原执行路径。
info!(
    todo_id = %todo.id,
    status = %todo.status,
    "Todo created successfully"
);

// ❌ 坏的日志：无上下文、无结构化参数。
info!("Todo created");

// ✅ 使用 span 关联上下文：
// span 会自动将上下文信息（如 request_id）带到 span 内的所有日志事件中。
let span = span!(Level::INFO, "create_todo", todo_id = %id);
let _guard = span.enter();
```

---

## 4. 敏感信息

- ❌ 不得记录密码、Token、密钥等敏感信息
- ❌ 不得记录完整的数据库连接字符串
- ✅ 可以记录脱敏后的调试信息（删除的参数值、截断的 Token）

---

## 5. 日志格式

开发环境输出人类可读格式（`tracing-subscriber` 的 `fmt` 层），
生产环境输出 JSON 格式以便日志收集系统解析。
