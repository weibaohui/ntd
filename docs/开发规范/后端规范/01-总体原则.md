# 后端规范 01：总体原则

> 定义 ntd 后端架构哲学与编码总则。

---

## 1. 架构原则

### 1.1 分层架构

ntd 后端采用经典三层架构 + Model 层：

```text
Handler（路由/HTTP 处理）
    ↓ 调用
Service（业务逻辑/编排）
    ↓ 调用
Repository（数据访问/SQL 封装）
    ↓ 使用
Model（Entity/DTO/Value Object）
```

- **Handler**：只做请求解析、参数校验、调用 Service、返回 Respone。不含业务逻辑。
- **Service**：纯业务逻辑，不感知 HTTP 细节（不引用 `axum::` 类型）。
- **Repository**：数据访问抽象，封装 SeaORM 查询，返回 Model 或 Option。
- **Model**：数据结构定义，含 Entity、DTO、枚举。

### 1.2 依赖方向

- Handler → Service → Repository → Model
- **禁止反向依赖**：Model 不得引用 Service，Service 不得引用 Handler

### 1.3 单一职责

- 每个 struct 只做一件事
- 每个函数控制在 30 行以内
- 超过两层嵌套循环/条件 → 抽成独立函数

## 2. 编码总则

### 2.1 零警告策略

```bash
cargo clippy --all-targets -- -D warnings
```

必须零告警零错误方可提交。

### 2.2 禁止 Unwrap

生产代码禁止 `.unwrap()` / `.expect()` / `panic!`（`#[cfg(test)]` 和 `build.rs` 除外）。

所有可能失败的操作必须使用 `?` 运算符传播错误，或使用 `Result` 类型显式处理。

### 2.3 函数长度

单函数体（不含函数签名、空行、注释）不得超过 **30 行**。

### 2.4 注释强制

- **逐行注释**：每行代码解释「为什么这么写」，而非「写了什么」
- **段落总览**：大段代码前先写一段注释说明整体思路
- **修改同步**：改动逻辑必须同步更新注释
- **删除标记**：确需删除注释时，用 `// 原注释内容（删除，待确认）` 标记

## 3. Conventions

### 3.1 命名

| 元素 | 规范 | 示例 |
|------|------|------|
| 模块 (mod) | snake_case | `user_service` |
| 结构体 | PascalCase | `UserService` |
| 枚举 | PascalCase | `TaskStatus` |
| 函数 | snake_case | `create_user` |
| 变量 | snake_case | `user_id` |
| 常量/静态 | SCREAMING_SNAKE | `MAX_RETRY_COUNT` |
| 错误类型 | PascalCase + Error 后缀 | `ValidationError` |
| Trait | PascalCase | `StorageBackend` |

### 3.2 文件组织

- 一个文件对应一个主要类型
- Handler 文件按资源分组：`user_handler.rs`、`todo_handler.rs`
- 测试文件放在 `backend/tests/` 集成测试目录，单元测试放在模块内 `#[cfg(test)]`
