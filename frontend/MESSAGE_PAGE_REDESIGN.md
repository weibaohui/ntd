# 消息页面改造方案

## 一、现状分析

### 1.1 当前问题

| 问题类型 | 描述 | 影响 |
|---------|------|------|
| **定位错位** | 左侧菜单叫"消息"，但页面实际是"消息配置中心" | 用户期望看到消息记录，却看到一堆配置表单 |
| **路径过长** | 查看消息记录需要：消息菜单 → 选择工作空间 → 点击Bot → 详情 → 展开消息记录（4步） | 高频操作变得繁琐 |
| **主次不分** | AgentPanel（Bot管理）、SlashCommands（斜杠命令）、Settings（设置）三个面板平铺 | 进入页面后找不到焦点 |
| **跨空间管理弱** | Bot只能在当前工作空间管理，无法全局查看和移动 | 需要跨空间管理Bot时操作不便 |

### 1.2 当前架构

```
MessagesPage（消息页面）
  └── WorkspaceMessageConfigPage（工作空间消息配置页）
        ├── WorkspaceAgentPanel（智能体管理面板）
        │     └── BotDetailPage（Bot详情页）
        │           ├── BotConfigCard（Bot配置）
        │           ├── PushStatusCard（推送配置）
        │           ├── WhitelistCard（白名单）
        │           └── HistoryTable（历史消息表格）
        ├── WorkspaceSlashCommandsPanel（斜杠命令面板）
        └── WorkspaceSettingsPanel（工作空间设置面板）
```

### 1.3 数据模型

- **消息记录**：存储在 `feishu_history_messages` 表，全局的，包含 `workspace_id` 字段
- **Bot**：`AgentBot` 表，`workspace_id` 表示当前服务的工作空间
- **斜杠命令**：`WorkspaceSlashCommand` 表，按工作空间隔离
- **默认响应设置**：`WorkspaceSettings` 表，按工作空间隔离

## 二、改造方案

### 2.1 核心思路

**拆分为两个独立页面**：

1. **消息监控台**（工作区级别）：专注展示当前工作空间的消息记录和快捷操作
2. **智能体管理中心**（全局级别）：统一管理所有Bot的绑定、配置、工作空间分配

### 2.2 左侧菜单调整

```
┌──────────────┐
│  工作区      │
│  ├─ 事项     │
│  ├─ 环路     │
│  ├─ 消息     │  ← 改造为「消息监控台」（工作空间级别）
│  ├─ 黑板     │
│  ├─ 仪表盘   │
│  └─ 看板     │
├──────────────┤
│  配置        │
│  ├─ 执行器   │
│  ├─ Skills   │
│  ├─ 工作空间 │
│  ├─ 智能体   │  ← 新增「智能体管理中心」（全局级别）
│  └─ 设置     │
└──────────────┘
```

| 接口 | 用途 |

## 七、设计原则

### 7.1 交互原则

1. **减少点击路径**：消息记录首屏可见，无需层层下钻
2. **视觉层级清晰**：消息记录为主，配置为次
3. **响应式适配**：桌面端三栏，移动端卡片列表
4. **即时反馈**：状态变更有明确的视觉反馈

### 7.2 代码规范

1. **组件拆分**：每个功能独立为组件，便于维护
2. **类型安全**：使用TypeScript严格类型检查
3. **状态管理**：合理使用React hooks管理状态
4. **错误处理**：所有异步操作有错误处理和用户提示
5. **性能优化**：消息列表支持虚拟滚动，避免长列表卡顿

## 八、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 消息记录量大 | 页面加载慢 | 分页加载 + 虚拟滚动 |
| 移动端适配复杂 | 布局错乱 | 采用Ant Design的响应式组件 |
| API不支持按workspace筛选 | 需前端过滤 | 确认后端接口能力，前端兜底 |
| 组件重构影响现有功能 | 回归测试覆盖 | 保留原有组件，逐步替换 |
